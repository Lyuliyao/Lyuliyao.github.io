Constructors

A constructor is a special member function that runs the moment an instance is created. Its one job: leave the instance in a valid, fully initialized state.

Part 1

Basic Constructors

The Problem: Uninitialized Instances

            
struct Time {
    int hour;
    int minute;
};

Time t;             // hour, minute are GARBAGE
std::cout << t.hour;     // undefined behavior
t.hour = 25;        // nothing stops an invalid time
  • Nothing forces t into a sensible state.
  • Reading t.hour before setting it is undefined behavior.
  • We want every Time to start out valid.

Hand-Initialization Does Not Scale

            
Time t;
t.hour = 9;         // every caller must remember
t.minute = 30;      // every field, every time

Time other;         // forgot to set minute -> bug
std::cout << other.minute;   // garbage
  • Setup is repeated at every creation site.
  • One forgotten line is a silent, hard-to-find defect.
  • The type should initialize itself.

What a Constructor Is

A constructor is a member function with the same name as the type and no return type. It runs automatically when an instance is created and is responsible for initializing the members.

A Default Constructor

            
struct Time {
    int hour;
    int minute;

    Time() {            // takes no arguments
        hour = 0;
        minute = 0;
    }
};

Time t;             // calls Time(): t is 00:00
  • No return type; name matches the type exactly.
  • A constructor that takes no arguments is the default constructor.
  • Time t; now produces a valid instance, automatically.

When the Constructor Runs

            
struct Time {
    int hour, minute;

    Time() {
        std::cout << "Time built" << std::endl;
        hour = 0; minute = 0;
    }
};

Time a;             // prints "Time built"
Time b;             // prints "Time built" again
  • You never call the constructor by name.
  • It fires once per instance, at the point of creation.
  • Two instances ⇒ two constructor runs.

Parameterized Constructors

            
struct Time {
    int hour;
    int minute;

    Time() { hour = 0; minute = 0; }

    Time(int h, int m) {        // parameterized
        hour = h;
        minute = m;
    }
};

Time start;            // 00:00  -> default
Time lunch(12, 30);    // 12:30  -> parameterized
  • Constructors can be overloaded like any function.
  • The compiler picks one by the construction arguments.
  • One type, several ways to build it.

A Constructor's Job: A Valid Initial State

A constructor should leave the instance in a state where every member function can be called safely. After Time lunch(12, 30);, both hour and minute hold meaningful values — never garbage.

Constructing Instances Several Ways

            
Time a;             // default ctor    -> 00:00
Time b(9, 15);      // (int,int) ctor  -> 09:15
Time c{9, 15};      // brace form      -> 09:15
Time d = Time(9,15);// build temp, then copy

Time e();           // NOT an instance: declares a function
  • Each line selects a constructor by its arguments.
  • Time e(); is the classic "most vexing parse" trap.
  • Use Time e; (no parentheses) for the default ctor.

The Constructor Sees the Members Directly

            
struct BankAccount {
    double balance;

    BankAccount(double initial) {
        if (initial < 0.0)
            balance = 0.0;      // refuse a bad start
        else
            balance = initial;
    }
};

BankAccount a(-40.0);   // balance becomes 0.00, not -40
  • The body is ordinary code that runs at creation.
  • It can reject bad input and still produce a valid instance.
  • That is the whole point: a guaranteed sane starting state.
Part 2

More on Constructors

The body works, but C++ gives us a more direct way to initialize members — and a few subtle rules to know about.

Assignment in the Body

            
struct Time {
    int hour;
    int minute;

    Time(int h, int m) {
        hour = h;        // ASSIGNMENT, not initialization
        minute = m;
    }
};
  • hour and minute are first default-built.
  • Then the body assigns over them — two steps.
  • It works for int, but it is not the preferred style.

The Member Initializer List

            
struct Time {
    int hour;
    int minute;

    Time(int h, int m) : hour(h), minute(m) { }
    //                  ^ initializer list, before the {}
};
  • The list appears after : and before the body.
  • hour(h) initializes hour directly — one step.
  • The body can be empty: all work is done in the list.

Initialization vs Assignment

A member in the initializer list is built once, with its final value. A member assigned in the body is built with a default value first, then overwritten. Initialization is the direct, preferred path.

const and Reference Members Force the List

            
struct Sensor {
    const int id;     // const: cannot be assigned later
    int& logRef;       // reference: must bind on creation

    Sensor(int identifier, int& log)
        : id(identifier), logRef(log) { }   // REQUIRED here
};
  • A const member cannot be assigned in the body.
  • A reference member must be bound when it is created.
  • Both must appear in the initializer list.

Order Follows Declaration Order

            
struct Range {
    int lo;
    int hi;
    int span;

    Range(int n) : lo(0), hi(n), span(hi - lo) { }
    // members build in DECLARATION order:
    // lo, then hi, then span
};
  • Members initialize in the order they are declared.
  • The order you write the list in does not change that.
  • List members in declaration order to avoid confusion.

Default Values in the Class

            
struct Time {
    int hour = 0;     // default member initializer
    int minute = 0;

    Time() { }                       // uses the defaults
    Time(int h) : hour(h) { }        // minute stays 0
};

Time a;        // 0:0
Time b(9);     // 9:0
  • A default member initializer gives a fallback value.
  • The initializer list overrides it when present.
  • Time(int h) sets hour; minute keeps its default 0.

Delegating Constructors

            
struct Time {
    int hour;
    int minute;

    Time(int h, int m) : hour(h), minute(m) { }
    Time(int h) : Time(h, 0) { }   // delegate to the 2-arg
    Time()       : Time(0, 0) { }  // delegate again
};
  • One constructor can call another in its initializer list.
  • Common construction logic lives in a single place.
  • No copy-pasted validation across three constructors.

The Implicitly-Generated Default Constructor

            
struct Point {
    int x;
    int y;
};                       // no constructor written

Point p;                 // OK: compiler-generated Point()
                         // x and y are uninitialized!
  • If you write no constructor, the compiler supplies one.
  • It default-builds each member — ints are left garbage.
  • Add default member initializers (int x = 0;) to be safe.

Declaring Any Constructor Removes It

            
struct Time {
    int hour, minute;

    Time(int h, int m) : hour(h), minute(m) { }
};

Time t;          // ERROR: no default constructor
Time u(9, 30);   // OK
  • Writing any constructor suppresses the generated default.
  • Time t; now fails to compile.
  • Add Time() = default; or write one to bring it back.

Lecture Summary

Main ideas

  • A constructor has the type's name and no return type.
  • It runs automatically when an instance is created.
  • Constructors can be overloaded; arguments pick one.
  • Initializer lists initialize members directly (one step).
  • Members build in declaration order, not list order.
  • const and reference members must be in the list.

Programming habits

  • Initialize every member; never leave garbage.
  • Prefer the initializer list over body assignment.
  • Delegate shared logic to one constructor.
  • Give members default initializers so instances start valid.

Review Questions

Core ideas

  • Q1. What return type does a constructor have?
  • Answer: None at all, not even void.
  • Q2. What is the default constructor?
  • Answer: The constructor callable with no arguments.
  • Q3. Where does the initializer list appear?
  • Answer: After the :, between the parameter list and the body.
  • Q4. Which members must be in the list?
  • Answer: const members and reference members.

Reasoning

  • Q5. In what order do members initialize?
  • Answer: Declaration order, regardless of the list's order.
  • Q6. Why is the list preferred over body assignment?
  • Answer: It initializes once with the final value; assignment builds, then overwrites.
  • Q7. What does a delegating constructor do?
  • Answer: Calls another constructor of the same type to share logic.
  • Q8. Why does Time t; fail after writing Time(int,int)?
  • Answer: Declaring any constructor removes the implicit default constructor.

Code Review

Spot the bug

  • Q1. What does span hold? 
                
struct Range {
    int span;
    int lo;
    int hi;

    Range(int n) : span(hi - lo), lo(0), hi(n) { }
};
Range r(10);   // r.span == ?
  • Answer: Garbage. Members build in declaration order: span first, so span(hi - lo) reads hi and lo before they are initialized. List order does not help — declare lo and hi before span.

Does it compile?

  • Q2. Build result? 
                
struct Sensor {
    const int id;

    Sensor(int i) { id = i; }
};
  • Answer: Build error: id is const and cannot be assigned in the body. Use Sensor(int i) : id(i) { }.
