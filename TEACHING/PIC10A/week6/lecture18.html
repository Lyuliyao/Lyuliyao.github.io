Function Overloading

C++ lets several functions share a name as long as their parameter lists differ. The compiler picks the best match for each call.

Chapter 14 overloading overload resolution implicit casts

Review

So far

  • One name, one function.
  • We chose between T, T&, const T&.

Today

  • Multiple functions can share a name.
  • The compiler picks one based on argument types.
  • Sometimes nothing is the "best" match: build error.

Roadmap

Allowed different parameters Same name, different parameter types is fine.
Not allowed only return type differs Same name and parameters, different return: build error.
Resolution best viable match Compiler scores candidates position by position.

Return Type Alone Is Not Enough

            
int    f() { return 0;   }
double f() { return 0.0; }   // build error
  • Same name, same (empty) parameter list, different return type.
  • The compiler cannot tell the two apart at a call site.
  • Parameter types must differ for overloading.

Order Matters

            
int    f(char c, bool b) { return 0;   }
double f(bool l, char r) { return 0.0; }   // OK
  • The two parameter lists are different.
  • (char, bool) is not the same as (bool, char).
  • This is legal function overloading.

Definition

It is legal to define functions with the same name and different parameter types. Doing so is called function overloading.

A Useful Example

            
void print(const vector<int>&    v);    // { 1, 2 }
void print(const vector<double>& v);    // (1.1 2.2)
void print(const vector<char>&   v);    // "12"

vector<int>    vi{1, 2};       print(vi);
vector<double> vd{1.1, 2.2};   print(vd);
vector<char>   vc{'1','2'};    print(vc);
  • One name, three implementations.
  • Each call resolves to the matching overload.
  • The reader sees a consistent interface.

Overload Resolution, Informally

The overload whose parameters match the arguments most closely is the one that is called.

Casting Can Step In

            
int    int_f(int    i) { return i; }
double dbl_f(double d) { return d + 0.5; }

int_f(2.3);   // 2.3 cast to int 2
dbl_f(4);     // 4   cast to double 4.0
  • When only one overload exists, casting glues the call together.
  • int ↔ double conversions are implicit.
  • This will matter when multiple overloads are candidates.

Best Match Position by Position

            
void f(int    a, int    b, double c);   // "i i d"
void f(int    a, double b, double c);   // "i d d"

f(1, 3,   5.6);   // i i d   -- exact in pos 2
f(1, 3.4, 5.6);   // i d d   -- exact in pos 2
f(1, 3,   5);     // i i d   -- best in pos 2 (no cast)
f(1.2, 3.4, 5.6); // i d d   -- best in pos 2 (no cast)
  • One position must be strictly better, others no worse.
  • Position 1 ties; position 3 ties; position 2 decides.

When There Is No Best

            
void g(int    a, int    b);   // "i i"
void g(double a, double b);   // "d d"

g(1, 3);       // i i, OK
g(1.2, 3.4);   // d d, OK

g(1, 3.4);     // build error
g(1.2, 3);     // build error
  • g(1, 3.4): (int,int) casts pos 2; (double,double) casts pos 1.
  • Each is worse than the other in some position.
  • No best match ⇒ build error.

The Tie-Break Rule

To regard one overload as better than another, it must be equal-or-better in every position and strictly better in at least one position. Otherwise the call is ambiguous and gives a build error.

References and const

            
void j(      vector<int>& p);   //       vector<int>&
void j(const vector<int>& p);   // const vector<int>&

      vector<int> v;    j(v);    // picks      vector<int>&
const vector<int> cv;   j(cv);   // picks const vector<int>&
  • For v, both overloads are viable; the non-const reference is the better match.
  • For cv, only the const overload is viable.
  • This explains why print(w) needs const vector<int>& when w is const.

Lecture 18 Summary

Main ideas

  • Overloads must differ in parameter types.
  • The compiler picks the best viable match.
  • "Best" means equal-or-better everywhere, strictly better somewhere.
  • No best ⇒ build error.

Programming habits

  • Overload only when the operations are conceptually the same.
  • Avoid overloads that differ only by implicit conversions.
  • Make const overloads available for const data.

Lecture 18 Review Questions

Core ideas

  • Q1. Can two overloads differ only by return type?
  • Answer: No, that is a build error.
  • Q2. Are (char,bool) and (bool,char) different parameter lists?
  • Answer: Yes; order matters.
  • Q3. What does the compiler do for an overloaded call?
  • Answer: Picks the best viable match by comparing parameter positions.
  • Q4. What happens when no candidate is best?
  • Answer: Ambiguous call; build error.

Reasoning

  • Q5. Why does g(1, 3.4) fail with g(int,int) and g(double,double)?
  • Answer: Each candidate casts in a different position; neither is strictly better.
  • Q6. When both j(vector&) and j(const vector&) exist, which is picked for a non-const argument?
  • Answer: The non-const reference overload (better match).
  • Q7. What if the argument is const?
  • Answer: Only the const overload is viable.
  • Q8. Why might f(int) and f(double) still work for f('X')?
  • Answer: char converts more readily to int than to double, so f(int) wins.

Lecture 18 Code Review

Trace the call

  • Q1. What prints? 
                
void f()                  { cout << '1' << endl; }
void f(char c)            { cout << '2' << c << endl; }
void f(bool b, char c)    { cout << '3' << b << endl; }

f('c', false);
  • Answer: Build error: no overload takes (char, bool); only (bool, char) exists, and the conversions in the wrong positions are not a unique best match.

Const overloads

  • Q2. What prints? 
                
void g(      string& s) { cout << '2' << endl; }
void g(const string& s) { cout << '3' << endl; }

      string s;
const string cs;
g(s);    // ?
g(cs);   // ?
  • Answer: 2 then 3. The non-const reference wins for s; only the const overload is viable for cs.
