References I

A reference does not store its own data. It is another name for a storage location someone else owns. We motivate it by trying, and failing, to write our own swap.

Chapter 10.1–10.3 bad_swap int& arrows

Review

Function calls so far

  • Each call introduces a function scope.
  • Arguments are copied into parameters at the start.
  • The body works with those copies.

The new question

  • std::swap(i, j) changes the caller's i and j.
  • How does that even work, given pass-by-value?
  • We need a parameter that is the argument, not a copy.

Roadmap

Attempt: bad_swap

            
void bad_swap(int a, int b) {
    int c = a;
    a = b;
    b = c;
}

int main() {
    int i = 0;
    int j = 1;
    cout << i << ' ' << j << endl;   // 0 1
    bad_swap(i, j);
    cout << i << ' ' << j << endl;   // 0 1
}
  • The body looks correct.
  • Yet i and j in main are unchanged.
  • Why? Because a and b are copies.

Why bad_swap Fails

What happens

  • Argument-to-parameter assignment copies the values.
  • The body swaps a and b — the copies.
  • Call ends: copies are destroyed; i and j were never touched.

What we need

  • A parameter that is the argument, not a copy.
  • Operations on the parameter must reach main's storage.

The Fix We Need

We need a kind of parameter whose creation does not copy. Instead, it should become another name for the argument variable. C++ calls this a reference.

Introducing int&

            
int main() {
    int i = 0;
    int& r = i;     // r has type int&: "reference to int"

    return 0;
}
  • int& is read aloud as "int ref".
  • r does not get its own storage.
  • r refers to the storage that i labels.

Definition

A reference does not store its own data. Instead it refers to already-existing data. Once initialized, it always refers to the same storage location — the arrow never moves.

Two Build Errors

            
int main() {
    int& r1;          // build error: must be initialized
    int& r2 = 4;      // build error: cannot bind to a literal

    int i = 0;
    int& r3 = i;      // OK
}
  • A reference must be initialized at its declaration.
  • It must refer to a real storage location, not a literal.
  • Literals like 4, 3.2, 'a', false have no address you can take.

Reading Through a Reference

            
int  i = 0;
int& r = i;

cout << i << ' ' << r << endl;   // 0 0

i = 1;
cout << i << ' ' << r << endl;   // 1 1
  • Printing r means: follow the arrow, print the box's value.
  • Updating i updates the box, so r "sees" it.
  • i and r are two names for the same data.

Writing Through a Reference

            
int  i = 1;
int& r = i;

r = 2;                // same effect as i = 2
cout << i << endl;    // 2
  • r = 2 follows the arrow and assigns to i's box.
  • Assigning to a reference changes the referenced data.
  • It does not redirect the arrow.

The Tempting Mistake: r = j

            
int  i = 2;
int& r = i;
int  j = 3;

r = j;                          // does NOT redirect r
cout << i << ' ' << j << endl; // 3 3, but r still points at i
  • Read this as: "follow r's arrow, store the value of j there."
  • It is the same as writing i = j.
  • r still references i — the arrow never moved.

Long Trace (Textbook Example 10.3.4)

            
int  i = 0;
int& r = i;
cout << i << ' ' << r << endl;            // 0 0

i = 1;
cout << i << ' ' << r << endl;            // 1 1

r = 2;
cout << i << ' ' << r << endl;            // 2 2

int j = 3;
cout << i << ' ' << r << ' ' << j << endl; // 2 2 3

r = j;
cout << i << ' ' << r << ' ' << j << endl; // 3 3 3

i = 4;
cout << i << ' ' << r << ' ' << j << endl; // 4 4 3

j = r;
cout << i << ' ' << r << ' ' << j << endl; // 4 4 4

j = 5;
cout << i << ' ' << r << ' ' << j << endl; // 4 4 5
  • Every line involving r behaves as if you had written i.
  • Lines that only touch j never affect i or r.
  • Whenever you are unsure: follow the arrow.

The Rule of Thumb

After int& r = i;, every later line involving r behaves as if you had written i instead.

What a Reference Is Not

Not a copy

  • Creating int& r = i does not allocate a new int.
  • There is one box, with two names.

Not reseatable

  • The arrow is fixed when you write = i.
  • Later r = j assigns through the arrow, never moves it.

Lecture 13 Summary

Main ideas

  • Pass-by-value copies, so bad_swap cannot reach the caller's variables.
  • int& is a reference: an arrow to an existing box.
  • A reference must be initialized and cannot bind to a literal.
  • Once initialized, the arrow never moves.

Programming habits

  • When in doubt, draw the boxes and arrows.
  • Read r = j as "follow r's arrow, store j's value there."
  • Use distinct names while learning so the picture is clear.

Lecture 13 Review Questions

Core ideas

  • Q1. What does int& mean?
  • Answer: The type "reference to int"; it refers to an existing int's storage.
  • Q2. Does a reference have its own storage?
  • Answer: No. It is another name for someone else's storage.
  • Q3. Can a reference be left uninitialized?
  • Answer: No — build error.
  • Q4. Why is int& r = 4; a build error?
  • Answer: A literal has no storage location to refer to.

Reasoning

  • Q5. Why does bad_swap fail?
  • Answer: Its parameters are copies of i and j.
  • Q6. Does r = j rebind r to j?
  • Answer: No, it assigns j's value into the box r already references.
  • Q7. After int& r = i; r = 2;, what is i?
  • Answer: 2.
  • Q8. What does "follow the arrow" mean?
  • Answer: Replace the reference with the variable it refers to before reasoning about the line.

Lecture 13 Code Review

Trace

  • Q1. What is printed? 
                
int  i = 10;
int& r = i;
r = r + 1;
i = i * 2;
cout << i << ' ' << r << endl;
  • Answer: 22 22. r = r + 1 means i = i + 1 ⇒ 11; i = i * 2 ⇒ 22; r still points at i.

Spot the error

  • Q2. Which lines fail to compile, and why? 
                
int  i = 0;
int& r1;            // (a)
int& r2 = 7;        // (b)
int& r3 = i;        // (c)
  • Answer: (a) uninitialized reference; (b) literal has no storage. (c) is fine.
