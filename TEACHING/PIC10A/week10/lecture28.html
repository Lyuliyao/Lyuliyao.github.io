C-Style Array

            
int a[4] = {10, 20, 30, 40};
cout << a[0] << endl;
cout << a[3] << endl;
  • C-style arrays have fixed size.
  • Indices still start at zero.
  • There is no bounds checking.

Looping Over an Array

            
int a[4] = {10, 20, 30, 40};

for (int i = 0; i < 4; ++i) {
    cout << a[i] << endl;
}
  • You must know or store the size.
  • The loop condition uses the array length.
  • i < 4 avoids one-past-the-end access.

Pointer Arithmetic

            
int a[4] = {10, 20, 30, 40};
int* p = a;

cout << *p << endl;
cout << *(p + 1) << endl;
  • An array name often behaves like a pointer to its first element.
  • p + 1 moves to the next int.
  • Indexing and pointer arithmetic are connected.

Bounds Bug: One Past the End

            
int a[3] = {4, 5, 6};

for (int i = 0; i <= 3; ++i) {
    cout << a[i] << endl;  // bug when i is 3
}
  • Valid indices are 0, 1, and 2.
  • i <= 3 eventually accesses a[3].
  • C-style arrays do not stop you from reading invalid memory.

Pointer Arithmetic Trace

            
int a[4] = {10, 20, 30, 40};

cout << *(a + 0) << endl;  // a[0]
cout << *(a + 2) << endl;  // a[2]
cout << 3[a] << endl;      // legal, but do not write this
  • a[i] is defined in terms of *(a + i).
  • Pointer arithmetic moves by elements, not raw bytes in your source code.
  • The strange 3[a] example explains the rule; it is not good style.

Lecture 28 Code Review

Trace it

  • Q1. What should you trace first? 
                
int a[5] = {};
a[0] = 7;
a[4] = 9;

cout << a[0] + a[4] << endl;
  • Answer: Trace which indices are explicitly changed and which remain zero-initialized.

Fix the design

  • Q2. What design question matters here? 
                
int a[3] = {1, 2, 3};
int* p = a + 1;

cout << *p << endl;
cout << *(p + 1) << endl;
  • Answer: p starts at a[1], so *(p + 1) reads a[2].
