Motivation: A struct's Invariant Can Be Broken

            
struct Time {
    int hour;     // intended: 0 <= hour < 24
    int minute;   // intended: 0 <= minute < 60
};

Time t;
t.hour = 99;      // nothing stops this
t.minute = -7;    // the object is now nonsense
  • With a struct, every member is writable by anyone.
  • External code can set values that violate our intent.
  • We need a way to hide the data and guard every change.

The class Keyword + public / private

            
class Time {
private:                 // reachable only inside Time
    int hour;
    int minute;
public:                  // reachable from any code
    Time(int h, int m) : hour(h), minute(m) { }
    int getHour() const { return hour; }
};

Time t(9, 5);
t.getHour();    // OK: public
t.hour = 20;    // ERROR: hour is private
  • class introduces a new type, like struct.
  • private members are reachable only from class members.
  • public members form the interface for outside code.

struct vs class Default Access

            
struct S { int x; };   // x is PUBLIC by default
class  C { int x; };   // x is PRIVATE by default

S s;  s.x = 1;         // OK
C c;  c.x = 1;         // ERROR: x is private
  • struct members default to public.
  • class members default to private.
  • Otherwise the machinery is identical — only the default differs.
  • Use class when you want encapsulation by default.

Encapsulation

Encapsulation: make member variables private and expose a small public interface. Because every change goes through code you wrote, the object can be kept valid — outside code can never reach in and corrupt it.

The Class Invariant

A class invariant is a condition every object of the class must always satisfy. For Time: 0 <= hour < 24 and 0 <= minute < 60. Encapsulation is what makes the invariant enforceable: private data + a guarded interface.

Enforcing the Invariant in the Constructor

            
class Time {
private:
    int hour;
    int minute;
public:
    Time(int h, int m) {
        if (h < 0 || h >= 24) h = 0;
        if (m < 0 || m >= 60) m = 0;
        hour = h;
        minute = m;
    }
};

Time bad(99, -7);   // becomes 00:00, still valid
  • The constructor establishes the invariant on creation.
  • Because hour is private, no object can skip this check.
  • Bad input is corrected, not stored.

Accessors (Getters)

            
class Time {
private:
    int hour, minute;
public:
    Time(int h, int m);
    int getHour()   const { return hour; }
    int getMinute() const { return minute; }
};

Time t(9, 5);
std::cout << t.getHour();    // 9, read-only access
  • Getters give controlled read access to private data.
  • They are const: reading never modifies the object.
  • Callers see the value but cannot bypass validation.

Mutators (Setters) with Validation

            
class Time {
private:
    int hour, minute;
public:
    bool setHour(int h) {
        if (h < 0 || h >= 24) return false;  // reject
        hour = h;
        return true;
    }
};

Time t(0, 0);
t.setHour(25);   // returns false, hour unchanged
t.setHour(14);   // returns true,  hour == 14
  • A setter is the only way to change a private member.
  • It rejects values that would break the invariant.
  • Returning bool tells the caller if it worked.

Defining Members Outside the Class

            
class Time {
private:
    int hour, minute;
public:
    Time(int h, int m);          // declared only
    int getHour() const;         // declared only
};

Time::Time(int h, int m) : hour(h), minute(m) { }
int  Time::getHour() const { return hour; }
  • Now we have a real class, the body can hold only declarations.
  • Time:: says "this function belongs to Time".
  • Inside Time::getHour we can name hour directly.

Scope Resolution, Precisely

ClassName::member is the scope resolution operator. It connects an out-of-class definition to its declaration inside the class body. The return type still comes first: int Time::getHour() const — int is the return type, Time:: names the owning class.

The Header File (.hpp)

            
// time.hpp
#ifndef TIME_HPP
#define TIME_HPP

class Time {
private:
    int hour;
    int minute;
public:
    Time(int h, int m);
    int getHour() const;
    bool setHour(int h);
};

#endif
  • The header holds the class definition + member declarations.
  • The include guard stops double-inclusion errors.
  • Anyone who #include "time.hpp" sees the interface.

The Source File (.cpp)

            
// time.cpp
#include "time.hpp"

Time::Time(int h, int m) : hour(h), minute(m) { }

int Time::getHour() const { return hour; }

bool Time::setHour(int h) {
    if (h < 0 || h >= 24) return false;
    hour = h;
    return true;
}
  • The .cpp holds member definitions, each with Time::.
  • It includes its own header so declarations match.
  • Interface (header) and implementation (cpp) are separated.

Worked Example: A Validated BankAccount

            
// account.hpp
class BankAccount {
private:
    double balance;          // invariant: balance >= 0
public:
    BankAccount(double b);
    double getBalance() const;
    bool withdraw(double amt);
    void deposit(double amt);
};

// account.cpp
BankAccount::BankAccount(double b)
    : balance(b < 0 ? 0 : b) { }
double BankAccount::getBalance() const { return balance; }
bool BankAccount::withdraw(double amt) {
    if (amt < 0 || amt > balance) return false;
    balance -= amt;
    return true;
}
void BankAccount::deposit(double amt) {
    if (amt > 0) balance += amt;
}
  • Private balance; public interface = the only access.
  • Invariant balance >= 0 established at construction.
  • Header = interface; cpp = BankAccount:: definitions.

Lecture 24 Summary

Main ideas

  • class introduces a type with public/private access.
  • class defaults to private; struct to public.
  • Encapsulation = private data + a guarded public interface.
  • That is what makes a class invariant enforceable.

Programming habits

  • Validate in constructors and setters, not at call sites.
  • Const getters in, validating mutators out.
  • Define members with ClassName::; keep the interface in the header, the work in the cpp.

Lecture 24 Review Questions

Core ideas

  • Q1. Default access for class vs struct?
  • Answer: class is private by default, struct public; otherwise identical.
  • Q2. What is encapsulation?
  • Answer: Private data behind a small, controlled public interface.
  • Q3. What does Time:: mean in int Time::getHour() const?
  • Answer: Scope resolution — this defines Time's declared getHour.

Reasoning

  • Q4. Why validate in the constructor?
  • Answer: So no object can ever begin life in an invalid state.
  • Q5. Why make balance private?
  • Answer: So all changes pass through validating functions; the invariant can't be bypassed.
  • Q6. What goes in the .hpp vs the .cpp?
  • Answer: Class definition + declarations + guard in the header; member definitions in the cpp.

Lecture 24 Code Review

Find the bug

  • Q1. Why won't this link/compile cleanly? 
                
// time.cpp
#include "time.hpp"
int getHour() const { return hour; }
  • Answer: Missing Time::. This defines a free function, not Time::getHour; hour is out of scope and const is invalid on a non-member. Write int Time::getHour() const.

Trace the result

  • Q2. What is the final balance? 
                
BankAccount a(-100);   // ctor: b < 0 ? 0 : b
a.deposit(40);
a.withdraw(200);       // amt > balance: refused
std::cout << a.getBalance();
  • Answer: 40. The constructor clamps -100 to 0, deposit adds 40, the over-limit withdraw is rejected.
