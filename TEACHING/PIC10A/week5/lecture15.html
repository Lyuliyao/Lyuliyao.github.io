Trace a Loop

            
int total = 0;
for (int i = 1; i <= 4; ++i) {
    total += i;
}
cout << total << endl;
  • Track i and total in a table.
  • Most loop bugs become visible after two iterations.
  • Always include the final failed condition in your trace.

Function Call Trace

            
int f(int x) {
    x += 2;
    return x;
}

int y = 5;
cout << f(y) << endl;
cout << y << endl;
  • Pass-by-value copies y into x.
  • The function returns 7.
  • The original y remains 5.

Reference Trace

            
void bump(int& n) {
    ++n;
}

int a = 4;
bump(a);
cout << a << endl;
  • Reference parameters can mutate caller variables.
  • Follow the alias arrow.
  • Use this only when mutation is intended.

Design Questions

Before writing code

Before debugging

  • What should happen on a tiny input?

  • Which operation may access invalid memory?

Review Drill: Trace Everything

            
int f(int x) {
    x *= 2;
    return x + 1;
}

int n = 3;
int answer = f(n);
cout << n << " " << answer << endl;
  • Trace the argument-to-parameter copy first.
  • n stays 3 because f receives a copy.
  • The returned value is 7, so the output is 3 7.

Review Drill: Mutation Through Reference

            
void add_tax(double& price) {
    price *= 1.095;
}

double total = 20.00;
add_tax(total);
cout << total << endl;
  • The parameter price references total.
  • The assignment inside the function changes the caller's variable.
  • This is exactly why non-const references must be used deliberately.

Lecture 15 Code Review

Trace it

  • Q1. What should you trace first? 
                
int product = 1;
for (int i = 2; i <= 4; ++i) {
    product *= i;
}
cout << product << endl;
  • Answer: Trace i and product; the final output is 24.

Fix the design

  • Q2. What design question matters here? 
                
void reset(int& x) {
    x = 0;
}

int n = 8;
reset(n);
cout << n << endl;
  • Answer: The parameter is a reference, so reset changes the caller's variable.
