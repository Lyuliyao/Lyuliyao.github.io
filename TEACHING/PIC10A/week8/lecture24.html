Initializer List

            
struct Pair {
    int first;
    int second;

    Pair(int a, int b)
        : first(a), second(b) {}
};
  • Initializer lists initialize members directly.
  • They are the preferred constructor style.
  • They become essential for const members and references.

Public and Private

            
class Counter {
private:
    int value;
public:
    Counter() : value(0) {}
    void bump() { ++value; }
    int get() const { return value; }
};
  • Private data protects representation.
  • Public functions form the interface.
  • Users should not depend on implementation details.

Outside Definitions

            
struct Point {
    double x;
    double y;
    double radius() const;
};

double Point::radius() const {
    return sqrt(x * x + y * y);
}
  • Declarations can live in the interface.
  • Definitions can be placed later or in cpp files.
  • Use ClassName:: to define member functions outside.

Design Questions

Before writing code

  • What data changes over time?
  • What function owns each task?
  • What type communicates the intent?

Before debugging

  • What should happen on a tiny input?
  • Where can scope or lifetime end?
  • Which operation may access invalid memory?

Class Interface

            
class BankAccount {
private:
    double balance;

public:
    BankAccount(double initial)
        : balance(initial) {}

    void deposit(double amount);
    double get_balance() const;
};
  • Private data prevents outside code from corrupting representation.
  • Public member functions define the allowed operations.
  • The constructor initializer list gives balance its starting value.

Definitions Outside the Class

            
void BankAccount::deposit(double amount) {
    if (amount > 0) {
        balance += amount;
    }
}

double BankAccount::get_balance() const {
    return balance;
}
  • BankAccount:: says which class the function belongs to.
  • Member definitions can access private member variables.
  • The const on get_balance must match the declaration.

public:
