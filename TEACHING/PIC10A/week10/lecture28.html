C-style Arrays I

Before std::vector there were raw arrays: a fixed block of elements laid out back to back in memory, with a size fixed at compile time and no safety net.

Declaring a Raw Array

            
int a[5];          // 5 ints, values uninitialized
double scores[3];  // 3 doubles
char grade[1];     // 1 char

a[0] = 10;
a[1] = 20;
  • T name[N] reserves room for exactly N elements.
  • The N must be a constant known at compile time.
  • Without an initializer the elements hold garbage.

Initializing an Array

            
int a[5] = {1, 2, 3, 4, 5};   // all five set
int b[5] = {1, 2};            // b = {1,2,0,0,0}
int c[5] = {};                // all zeros
int d[]  = {7, 8, 9};         // size deduced: 3
  • A brace list initializes element by element.
  • Missing trailing values are zero-initialized.
  • Omitting the size lets the compiler count the list.

Indexing Starts at Zero

            
int a[5] = {10, 20, 30, 40, 50};

a[0];   // 10  — first element
a[4];   // 50  — last valid element
a[2] = 99;     // a = {10,20,99,40,50}
  • Valid indices for a[5] are 0 through 4.
  • The last element is at index N - 1, never N.
  • Element i sits i slots past the start.

No Bounds Checking

            
int a[5] = {1, 2, 3, 4, 5};

a[5]  = 100;   // out of range — UNDEFINED BEHAVIOR
a[-1] = 7;     // out of range — UNDEFINED BEHAVIOR
int x = a[99]; // reads some unrelated memory
  • C++ does not check that the index is in range.
  • An out-of-range access is undefined behavior.
  • It may crash, corrupt data, or appear to "work".

Undefined Behavior, Stated Plainly

Writing past the end of a raw array does not throw and does not warn. The program may keep running with silently corrupted memory — the worst kind of bug to chase.

The Size Is Fixed Forever

            
int n = 5;
int a[n];          // not standard C++ (size must be constant)

const int N = 5;
int b[N];          // OK — N is a compile-time constant

// b cannot grow: there is no b.push_back(...)
  • The size is baked in when the array is created.
  • You cannot resize, append, or shrink it.
  • The size must be a compile-time constant.

Traversing With a Size Variable

            
const int N = 5;
int a[N] = {3, 1, 4, 1, 5};

int sum = 0;
for (int i = 0; i < N; ++i) {
    sum += a[i];
}
cout << sum << endl;   // 14
  • An array does not know its own length.
  • You must carry the size N alongside it.
  • The loop condition is i < N, never i <= N.

The Classic Off-by-One

            
const int N = 5;
int a[N] = {1, 2, 3, 4, 5};

for (int i = 0; i <= N; ++i) {   // BUG: i == 5 is out of range
    cout << a[i] << ' ';
}
  • i <= N reaches a[5], which does not exist.
  • That last read is undefined behavior.
  • Always loop while i < N.

Arrays vs std::vector

Size fixed vs dynamic Array size is locked; a vector can grow and shrink.
Length tracked or not A vector knows .size(); an array does not.
Safety .at() vs nothing v.at(i) can range-check; a[i] never does.

Prefer vector by Default

            
// Raw array: size and data travel separately
int a[5] = {1, 2, 3, 4, 5};

// vector: size travels with the data, can grow
std::vector<int> v = {1, 2, 3, 4, 5};
v.push_back(6);
cout << v.size() << endl;   // 6
  • vector carries its own length and resizes.
  • It copies cleanly and can be returned from a function.
  • Use raw arrays only when an interface forces them.

Passing an Array to a Function

            
int sum(int a[], int n) {       // a is really int*
    int s = 0;
    for (int i = 0; i < n; ++i) s += a[i];
    return s;
}

int data[4] = {2, 4, 6, 8};
cout << sum(data, 4) << endl;    // 20 — size passed by hand
  • An array argument "decays" to a pointer to its first element.
  • The function never receives the length.
  • So you must pass the size as a separate parameter.

Decay Loses the Size

Inside the function, the parameter is a pointer — not an array. sizeof on it gives the size of a pointer, not the array. The length only survives if you pass it yourself. (Detail next lecture.)

Lecture 28 Summary

Main ideas

  • T a[N] is a fixed block of N elements.
  • Indices run 0 to N-1; no bounds checking.
  • Out-of-range access is undefined behavior.
  • The size is a compile-time constant and cannot change.

Programming habits

  • Loop while i < N, never i <= N.
  • Pass the size alongside the array to functions.
  • Reach for std::vector unless an interface forbids it.

Lecture 28 Review Questions

Core ideas

  • Q1. What are the valid indices of int a[6];?
  • Answer: 0 through 5.
  • Q2. What does int b[5] = {1, 2}; store?
  • Answer: {1, 2, 0, 0, 0} — missing values are zero.
  • Q3. Can a raw array grow at runtime?
  • Answer: No; its size is fixed at compile time.

Reasoning

  • Q4. What happens when you read a[N] for int a[N]?
  • Answer: Undefined behavior — out of range, no diagnostic.
  • Q5. Why pass a size parameter to sum(int a[], int n)?
  • Answer: The array decays to a pointer and loses its length.
  • Q6. Why is std::vector usually preferred?
  • Answer: It tracks its own size, resizes, and copies safely.

Lecture 28 Code Review

Spot the bug

  • Q1. What is wrong here? 
                
int a[3] = {1, 2, 3};
for (int i = 1; i <= 3; ++i)
    cout << a[i] << ' ';
  • Answer: It skips a[0] and reads a[3] (out of range). Use for (int i = 0; i < 3; ++i).

What prints?

  • Q2. Output? 
                
int a[5] = {4};
int s = 0;
for (int i = 0; i < 5; ++i) s += a[i];
cout << s << endl;
  • Answer: 4. Only a[0] is 4; the rest are zero-initialized.
