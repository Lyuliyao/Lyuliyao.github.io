From bad_swap to good_swap

            
void good_swap(int& a, int& b) {
    int c = a;
    a = b;
    b = c;
}

int main() {
    int i = 0;
    int j = 1;
    cout << i << ' ' << j << endl;   // 0 1
    good_swap(i, j);
    cout << i << ' ' << j << endl;   // 1 0
}
  • The body is identical to bad_swap.
  • Only the parameter types changed: int → int&.
  • Now the call actually swaps i and j.

What the Call Does

            
// good_swap(i, j) is, conceptually:
{
    int& a = i;     // arrow from a to i
    int& b = j;     // arrow from b to j

    int c = a;      // c gets value at a's arrow target
    a = b;
    b = c;
}
  • Argument-to-parameter assignment for a reference parameter does not copy.
  • It tells the parameter what location to refer to.
  • Now a and b are arrows into main's scope.

Trace good_swap(i, j)

Setup

  • i = 0, j = 1 in main.
  • a references i; b references j.

Execute the body

  • c = a: follow a ⇒ c = 0.
  • a = b: follow both ⇒ i = j ⇒ i becomes 1.
  • b = c: follow b ⇒ j = 0.
  • End of scope: a, b, c vanish; i and j are swapped.

Why It Works

For a reference parameter, the argument-to-parameter assignment installs an arrow into the caller's scope. Every operation on the parameter is — by following the arrow — an operation on the caller's variable.

Mutation Is Now a Choice

            
void by_value(int x) { x = 100; }     // cannot affect caller
void by_ref  (int& x) { x = 100; }    // does affect caller

int n = 3;
by_value(n);  cout << n << endl;       // 3
by_ref  (n);  cout << n << endl;       // 100
  • int x: copy in, no caller effect.
  • int& x: arrow in, mutation reaches the caller.
  • The signature tells the reader which kind of call this is.

References to const

            
int main() {
    int        i = 0;
    const int  ci = 1;

    int&        r1 = i;        // OK
    const int& rc1 = ci;       // OK
    const int& rc2 = i;        // OK -- read-only view of i
}
  • const T& is "reference to const T".
  • It can refer to a non-const variable.
  • While referring, it treats the location as if it were const.

Build Error 10.5.2: No Writing

            
int        i = 0;
const int& rc = i;

// rc = 3;        // build error: cannot assign through ref-to-const

i = 5;            // OK -- i is not const
cout << rc << endl;   // 5  (rc still sees changes)
  • You may not assign through rc.
  • The underlying variable can still change by other means.
  • rc sees those changes; it just refuses to make them.

Build Error 10.5.3: No Loophole

            
const int ci = 1;

int& r = ci;     // build error: non-const ref to const variable
  • If this were allowed, r = 99 would change a const variable.
  • That would break the contract ci made when declared const.
  • The compiler refuses to let us punch a hole in const.

The Two Rules Side by Side

T&

  • May refer to non-const variables only.
  • May write through.
  • Cannot bind to a const target.

const T&

  • May refer to const or non-const variables.
  • May not write through.
  • The polite arrow.

Long Trace (Textbook Example 10.5.4)

            
int        i = 0;
const int  ci = 1;

int&        r1  = i;
const int& rc1 = ci;
const int& rc2 = i;

cout << i << ' ' << r1 << ' ' << rc2 << endl;   // 0 0 0

i = 5;
cout << i << ' ' << r1 << ' ' << rc2 << endl;   // 5 5 5

r1 = 6;
cout << i << ' ' << r1 << ' ' << rc2 << endl;   // 6 6 6
  • r1 and rc2 both point at i.
  • Updates to i are visible through both arrows.
  • rc2 cannot itself perform an update; r1 can.

References Are Already Const

Even an ordinary T& is "const" in one sense: the arrow itself never moves. The word const in const T& refers to the target — you cannot write through the arrow.

Lecture 14 Summary

Main ideas

  • A reference parameter installs an arrow into the caller's scope.
  • That is what lets good_swap actually swap.
  • const T& is a read-only arrow.
  • Two build errors enforce const: cannot write through, cannot bypass with a non-const arrow.

Programming habits

  • Use a reference parameter only when mutation is intended.
  • Use const T& when you want a no-copy view that promises not to write.
  • When tracing, draw the arrows across scopes.

Lecture 14 Review Questions

Core ideas

  • Q1. What does int& a mean as a parameter?
  • Answer: An arrow installed to the argument's storage; no copy is made.
  • Q2. Why does good_swap succeed where bad_swap failed?
  • Answer: Operations on a, b are operations on the caller's i, j.
  • Q3. What does const int& rc mean?
  • Answer: A read-only arrow to an int; you may not assign through rc.
  • Q4. Can const int& rc = i; when i is non-const?
  • Answer: Yes; rc just promises not to write.

Reasoning

  • Q5. Why is int& r = ci; a build error?
  • Answer: It would let you write to ci through r, breaking const.
  • Q6. If i changes, does const int& rc = i see the change?
  • Answer: Yes — const blocks writing, not reading.
  • Q7. Does good_swap(i, j) copy any ints?
  • Answer: Only the local c inside the body; the parameters are arrows.
  • Q8. What does the const in const T& qualify?
  • Answer: The target of the arrow, not the arrow itself.

Lecture 14 Code Review

Trace through the call

  • Q1. What does this print? 
                
void f(int& a, int b) {
    a = a + b;
    b = b + 1;
}
int x = 10, y = 3;
f(x, y);
cout << x << ' ' << y << endl;
  • Answer: 13 3. a is an arrow to x, so x becomes 13; b is a copy of y, so y is unchanged.

Spot the const error

  • Q2. Which lines fail to compile? 
                
const int ci = 7;
      int  i  = 8;

int&        a = ci;   // (a)
const int& b = ci;   // (b)
const int& c = i;    // (c)
  • Answer: Only (a) fails — a non-const reference cannot bind to a const variable. (b) and (c) are fine.
